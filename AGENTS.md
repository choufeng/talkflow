# TalkFlow 派规则设定

## 第一条铁律：纯函数抽象

**所有可抽象为纯函数的数据方法均应抽象，且本项目应以函数式编程方式构建。**

### 具体准则

1. **数据转换**：任何从输入数据到输出数据的转换逻辑，若不依赖外部状态、不产生副作用，必须抽象为纯函数（pure function）。

2. **副作用隔离**：I/O、网络请求、UI 更新、状态变更等有副作用的操作应与纯数据逻辑严格分离，放置在专门的边界层（如 `sideEffects/`、`IO/` 或类似模块）中。

3. **不可变数据**：优先使用值类型（`struct`）和不可变状态，避免 `class` 引用类型带来的共享可变状态。

4. **组合优于继承**：通过函数组合和高阶函数构建复杂逻辑，避免深层类继承。

5. **类型驱动**：利用 Swift 强类型系统，通过类型明确表达函数的输入输出契约，减少运行时错误。

6. **管道式流程处理**：流程化的操作应使用函数式编程中的管道（pipeline）模式——将复杂流程拆解为一系列独立的纯函数步骤，通过组合符号（如 `|>` 概念或链式调用）串联执行，每一步的输出即为下一步的输入。管道应具备：
   - 每个步骤是独立可测试的纯函数
   - 步骤间无共享状态，仅通过数据传递
   - 步骤顺序可灵活重组、插入或移除
   - 错误沿管道传播，不在步骤内部吞没

7. **声明式优先**：描述"要什么"而非"怎么做"。用 `filter { $0.isActive }` 表达意图，而非手写 for 循环加 if 判断。声明式代码更贴近需求描述，易读、易维护、不易出错。

8. **高阶函数优先**：函数是一等公民，应频繁使用接收函数或返回函数的模式。不仅是 `map`/`filter`/`reduce`，还应自定义高阶函数替代模板代码和重复逻辑。高阶函数是消除重复的最强工具。

9. **柯里化与部分应用**：将多参数函数拆解为单参数函数链（`(A, B) -> C` → `(A) -> (B) -> C`），每个步骤关注单一职责。部分应用可固化上下文参数，生成专用函数，极大提升复用性和可测试性。

10. **Monadic 链式处理**：对带上下文的计算（可选值 `Optional`、可能失败 `Result`、异步 `Future`、集合 `Array`）统一使用 `map` / `flatMap` / `bind` 串联，而非层层 if-let 解包或 try-catch 嵌套。链式处理让"上下文穿越"变得透明。

11. **代数数据类型（ADT）**：用 sum type（枚举带关联值）和 product type（struct）精确建模领域状态，让非法状态不可表示——编译器即可消除一整类 bug，根本不需要运行时防御。

12. **模式匹配与穷尽性**：对所有 sum type 使用穷尽模式匹配（switch），编译器会检查是否有遗漏分支。替代运行时的 if-else 链和不安全的类型转换，将"遗漏的处理"从运行时错误变为编译错误。

13. **引用透明性**：任何函数调用都可以被替换为它的返回值而不改变程序行为。违反引用透明性的地方必须明确标记（如注释标注包含副作用），让副作用在代码中"可见"。

14. **惰性求值**：计算应延迟到真正需要结果时再执行，尤其处理大数据集或昂贵计算。用 `lazy` 序列、generator 模式推迟计算，避免不必要的中间分配和过早求值。

15. **递归代替循环**：遍历和状态累积优先用递归（尤其尾递归）表达。递归的函数式结构消除了循环变量、索引越界和可变累加器，更清晰、更易证明正确性。

16. **构造与副作用分离**：对象的创建/初始化不应触发任何副作用（网络请求、文件读写、UI 操作）。"初始化一个值"应该是纯净的操作，副作用在显式调用时执行。

17. **在 Swift 中的实践**：
   - 优先使用 `struct` + `enum` 建模数据
   - 数据转换逻辑放在独立的纯函数中，不在类型方法中混入副作用
   - 使用 `map`、`flatMap`、`compactMap`、`filter`、`reduce` 等高阶函数处理集合
   - 构建管道式的自定义操作符（如 `|>` 或通过链式方法）串联数据处理步骤
   - 用 `throw` / `Result` 表达可失败操作，避免返回可选值后丢失错误信息
   - 用 `lazy` 修饰集合链式调用，避免中间分配
   - 初始化器只做赋值，不做 I/O
   - 副作用操作通过协议抽象，便于测试和替换
